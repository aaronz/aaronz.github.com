不强求跨浏览器显示一致

  • 网站内容第一，形式第二。
  • 首先确定跨浏览器内容的一致性作为基线，然后逐步优化形式使其趋于一致。
  • 由于浏览器差异性确实存在，不必强求显示效果完全一致。

使用稳定模板库简化编程

  • 稳定的模板库可以简化跨浏览器支持工作，很多规则都在模板库中实现。
  • 例如Html5Boilerplate/Bootstrap就是不错的选择，它们都提供了跨浏览器的兼容性支持。

基于稳定的Web标准编程

  • 尽量避免使用浏览器的特有功能，这些功能的使用是跨浏览器实现的一大障碍。
  • 新的Web标准通常会包含很酷的功能，但是新标准和可能会经过一系列更改变为稳定版本，因此考虑跨浏览器网站实现注意对新标准采取保守态度，尽量基于稳定的标准编程。

  • 我们无法预知客户通过什么浏览器在访问网站，例如客户浏览器可能不支持flash，不支持JavaScript或者特殊的CSS。网站在这种极端情况下也需要能够正常输出必要的内容。

增强保护模式(EPM) – IE10+

  • 增强保护模式 (EPM) 是指一系列的安全改进措施：
  • 64 位选项卡进程，让基于内存的安全措施更有效。
  • 对个人和计算机配置资源的访问限制。
  • 对企业和其他本地资源的访问限制。
  • EPM有效地在不可信内容和敏感系统数据之间创建了一个沙箱，使恶意行为更加难以执行，并且降低其影响。

可互操作的Quirks模式 – IE10

  • 在页面顶部添加下面的 meta 标记，以选择启用传统的 Quirks 行为:
  • <meta http-equiv="X-UA-Compatible" content="IE=5">

拼写检查和自动更正支持 – IE10+

  • IE10 支持拼写检查和自动更正。spellcheck 属性可用于控制拼写检查。spellcheck  属性报告是否对文档对象模型 (DOM) 子树启用拼写检查。
  • 在某些情况下，IE10内置的拼写检查可能与网站预先存在的拼写检查功能产生冲突。内置的拼写检查菜单优先于网站预先提供的上下文菜单。
  • 如果网站受到影响，检测是否存在 spellcheck 属性。如果存在，则为该实例禁用自定义拼写检查功能： 
    •  if (document.documentElement.spellcheck == undefined)
  • http://ie.microsoft.com/testdrive/Browser/SpellChecking/

弃用文档模式 – IE11+

  • IE8 中引入了文档模式，将网站从不同版本的专有功能过渡到新版本的标准功能。从IE11 开始，文档模式被视为已弃用。
  • <!DOCTYPE html>

伪点击测试在IE11边缘模式禁用 – IE11+

  • IE10及以前版本通过透明背景允许鼠标单击和触摸交互。此功能称为伪点击测试。
  • 因为伪点击测试根据透明层背后的内容而有所变化，其测试效果并不一致，并且没有用于方向控制操作（如平移或缩放）。
  • 为了更好地兼容其他浏览器以及为所有交互事件（包括触摸和缩放）提供更加一致的体验，在 IE11 边缘模式中禁用了伪装点击测试。
  • 其影响在于某些处于透明层背后的元素不响应点击事件。可以通过z-index调整目标层次。
  • http://webdemos.azurewebsites.net/hit-testing.html

getElementById – IE8+

  • IE7及以前版本getElementById 首先返回name=id的元素。
  • IE8之后getElementById不再按name属性查找，并且对于输入参数大小写敏感。
    • var x = document.getElementById('test')

    • document.getElementById('test').hasAttribute('testAttribute')

事件对象扩展冲突 – IE8+

  • IE8对添加了新的事件属性，如果代码中存在为事件添加的同名属性，则会造成冲突导致脚本错误。
  • 解决方案是将现有的冲突属性重新命名。
    • // Read-only in IE8
event.source = myObject;
// change source to mySource
event.mySource = myObject;

attributes排序差异 – IE8+

  • IE8 DOM属性(attributes)排序较之前版本有所变化，改变化会影响attributes[]数组元素顺序以及innerHTML和outerHTML输出。
  • 解决方案是修改代码逻辑使其不要依赖于属性顺序。
    • // May differ in IE8
attr = elm.attributes[1];
// Use attribute name instead
attr = elm.attributes["id"];

从DOM删除IFrame释放资源 – IE9+

MIME 类型和纯文本内容 – IE9+

  •  IE9 标准模式 “text/plain” MIME 类型的文档不会通过 MIME 探查为其他类型。文档仅以纯文本方式呈现或下载。
  • 这样可以更轻松地共享 HTML 源代码片段。防止”text/plain”类型的脚本注入攻击。
  • 配置服务器所有文档发送适当的 Content-Type 标头。例如，如果你的服务器提供可移植文档格式 (PDF) 文件以供下载，请确保文件使用 “application/pdf” MIME 类型。

原生XML对象类型 – IE9+

  • 站点中的代码尝试混合这两种对象时，通常会引发 JavaScript 异常，这可能导致兼容性问题。
  • 例如： xhr.responseXML就是一个MSXML对象

对象元素回退适用于DOM – IE9+

  • object 元素含回退内容（通常是 embed 元素）时，IE9 会解析此内容，并且将其包含在文档对象模型 (DOM) 中，之前版本的IE不会执行此操作。
  • 如果 object 元素与其任意一个回退元素具有相同的名称属性，则 window["myName"] 现在会返回所有具有名称 "myName" 的元素的集合。
  • 在访问关于返回值的方法和属性时，假定IE会返回单个元素（通常是 object 元素）的页面可能导致出现异常。
  • 使用document["myName"]可以正常返回object对象。

减少 MIME 类型的安全风险 – IE9+

  • 如果服务器发送响应头 "X-Content-Type-Options: nosniff"，则 script 和 styleSheet 元素会拒绝包含错误的 MIME 类型的响应。这是一种安全功能，有助于防止基于 MIME 类型混淆的攻击。

脚本元素和事件执行 – IE9+

  • IE9 标准模式为 script 元素引入了基于标准的 load 事件。以前版本的 IE仅支持 script 元素的不可互操作的 onreadystatechange 事件。
  • 通常这些事件用于页面加载以后执行，以防止阻碍页面加载。在这些情况下，同时为 onreadystatechange 和 load 注册可能导致回调执行两次。这通常导致脚本错误或页面功能损坏。
  • 建议对这些情形使用基于标准的 load 事件。Web 开发人员应使用功能检测来检测支持 script 元素的 load 事件的浏览器。在传统版IE中，应回滚脚本以使用 onreadystatechange。

利用Mutation Observer – IE11+

  • IE11 中的Mutation Observer改进了对突变事件和突变属性的支持。
  • DOM3的支持方式基于DOM事件，存在性能问题。DOM4的支持方式取消对DOM事件的依赖，提升性能。
    • 
// DOM3
document.body.addEventListener("DOMNodeInserted", nodeAddedCallback, false);
document.body.addEventListener("DOMNodeRemoved", nodeRemovedCallback, false);
// DOM4
new MutationObserver(nodesAddedAndRemovedCallback).observe(document.body,{ childList: true, subtree: true });

JavaScript兼容性更改

数组索引处理 – IE9+

  • IE8在数组索引处理方面不符合 ECMAScript（第三方版本）规范。在创建索引大于2147483647的Array元素时，创建的新元素的索引将是一个负整数。
  • IE9正确处理了使用 2E+31-1 与 2E+32-2 之间的索引的 Array 元素。IE8行为不会在任何 IE9文档模式中复制。
  • http://jsfiddle.net/aaronzhcl/K73Gy/
  • http://webdemos.azurewebsites.net/array-index.html

枚举JavaScript属性 – IE9+

  • IE9对JavaScript对象模型有所更改，使得JavaScript属性的枚举方式与较早版本中的枚举方式不同。
  • 使用 for…in 语句时，属性枚举的顺序在任何文档模式中都可能与IE8返回的顺序不同。
  • http://jsfiddle.net/aaronzhcl/LjrQz/
  • http://webdemos.azurewebsites.net/enum-property.html

函数指针方法调用 – IE9+

  • IE9以前版本支持将方法的指针进行缓存并随后使用缓存的指针来调用方法。自IE9开始，取消了这项支持以改善与其他浏览器的互操作性。 IE9需要有一个对象才能调用方法。默认情况下会作用于window对象。改进后有以下两种调用方式，
  • 使用call方法（所有函数的一个属性）显式提供适当的调用对象
  • 使用JavaScript的bind API将隐式调用对象与该方法关联
  • http://jsfiddle.net/aaronzhcl/D789Q/
  • http://webdemos.azurewebsites.net/method-pointer.html

间接eval调用作用域 – Ie9+

  • IE9以前版本，传递给间接eval的字符串将在本地函数作用域内求值。
  • 从IE9标准模式开始，该字符串根据ECMAScript语言规范第5版的规定在全局作用域中求值。
  • http://jsfiddle.net/aaronzhcl/v63ZL/

数字精度和SSE2差异 – IE9+

  • IE9在平台支持的情况下会使用Streaming SIMD Extensions 2 (SSE2)来提高数学运算速度和精度，因此会获得和IE8及以前版本不同的精度。
  • http://jsfiddle.net/aaronzhcl/GsK8T/

Null协议返回值 – IE9+

  • IE9在处理返回"null"值的JavaScript时遵循以下HTML5规定。浏览器必须将URL视为已经返回HTTP 204 无内容，其中不得包含响应正文。
  • 由于作为JavaScript协议(javascript:)一部分执行返回"null"，IE9会将URL视为其返回了“HTTP 204 无内容”，因此iframe为空，无论JavaScript协议中运行其他什么JavaScript。
  • http://jsfiddle.net/aaronzhcl/7j4xa/

夏令时处理 – IE10+

  • IE9和以前版本中，日期通过应用ECMAScript规范中来存储夏令时调整时间。为提高准确性，尤其是过去日期（历史数据）的准确性，IE10依 据系统规则存储夏令时调整时间。如果你的代码在你的Web应用程序中计算历史日期，或具有自定义逻辑可解决浏览器日期计算不准确的问题，要确保在升级 Web应用程序使其适用于IE10时，自定义逻辑仍可正常使用。
  • 对于夏令时转换发生在午夜（将时钟回拨）的时区，系统时间实际在过渡边界前1毫秒(ms)进行转换。通过在过渡边界前1ms进行转换，Windows 7及以上版本将仍处于夏令时转换的当天，但会在夏令时转换完成后的状态下向后回拨时钟。
  • http://jsfiddle.net/aaronzhcl/57wsF/

验证JavaScript框架对新版IE的支持

  • 许多站点仍在使用与新版本的IE不兼容的旧版JavaScript框架。许多现有JavaScript框架包含的功能取决于现有IE特定的行为或quirks模式。因此，在IE中所作的更改可能导致许多受欢迎的JavaScript框架部分无法正确工作。
  • 为做演示，以下为需要更新以支持IE9的受欢迎 JavaScript 框架的列表。
  • Cufon 1.09i+
  • jQuery 1.5.1+
  • jQuery UI 1.6.8+
  • MooTools 1.3+
  • Prototype 1.7+

CSS 和布局兼容性更改

CSS表达式 – IE8+

  • CSS表达式的支持从IE8版本移除。
  • background-color: expression( (new Date()).getHours()%2 ? "#000" : "#fff" );
  • 可以换用脚本控制
    • 
        var elm = document.getElementById("main");
        if ((new Date()).getHours() % 2) {
            elm.style.backgroundColor = "#000";
        } else {
            elm.style.backgroundColor = "#fff";
        }
  • http://webdemos.azurewebsites.net/css-expression.html

泰语和东亚文字的字体大小 – IE9+

  • 在 IE9 中，泰语和东亚文本看起来比在 IE8 及更早版本中小一些。
  • 在IE8 中，呈现泰语和东亚文本所用的字体大于在以下情况下指定的字体：
  • 指定的字体大小为 9pt 或更小
  • 指定的字体系列不支持泰语或东亚字符，例如 Arial
  • 在这种情况下，以 8pt 的 Arial 字体表示的泰语段落会以回退字体呈现，并且稍后会放大该字体以匹配 Arial 指标。因此，如果 Web 作者要求 8pt 的这类字体，则字体的实际大小会大于其预计的大小。
  • 在IE9 中，始终会考虑指定的字体大小。因此，由于不会再放大回退字体，文本看起来就会小一些。

圆角和传统版渐变筛选器 – IE9+

  • 使用传统版渐变筛选器填充已使用 border-radius 属性生成圆角的元素时，渐变会超出元素的边框。
  • http://jsfiddle.net/aaronzhcl/7quPX/

文本布局和自然指标 – IE9+

  • 对于 IE9 标准模式（以及更高版本）中的文本布局，IE9 使用自然指标，而非其他 Windows 浏览器使用的 Windows 图形设备接口 (GDI) 指标。GDI 指标将文本与像素边界对齐，但自然指标使用像素间的间距创建更准确地呈现和可读的文本。
  • 为早期文档模式编写的页面布局在IE9 中可能无法正确显示。最常见的错误是意外的文本换行，换行文本下的元素可能会被覆盖。当文本框未包括额外的水平空间或者文本框的大小连接到页面中的其他元素（如图形）时，最有可能出现此错误。

复选框和单选按钮布局默认值 – IE11+

  • 在 IE11 模式中，margin、padding 和 box-sizing 的默认值针对 checkbox 和 radio button 输入类型将进行更新。
  • padding 的默认值现在是 0px。
  • margin 的默认值现在是 3px，但是 marginLeft 除外，后者的默认值是 4px。
  • box-sizing 设置为 "border-box"。

默认字体更改 – IE11+

  • 我们为使用 JavaScript 的几种语言的 Windows 应用商店应用更新文档对象模型 (DOM) 默认字体和 UI 字体。 这意味着取决于IE的原默认字体的某些网页和 Web 应用看起来可能不同。
  • 尽管应该改进了它们的整体外观，但此更改会困扰某些取决于某些脚本特定外观的测试。例如，没有字体规范的阿拉伯语现在默认为“Times New Roman”而非“Simplified Arabic”。

弹性框(Flexbox)布局更新 – IE11+

  • 2012 年 9 月，CSS 弹性框（“Flexbox”）布局模块作为 W3C 候选推荐发布，其中合并了对该规范某些方面的更改。 IE11 利用这些更改更新了弹性框支持，包括重命名的属性、重命名的属性值等。 随着不断向前发展，将使用新的无前缀名称（和值），并相应地更新现有站点。

传统功能支持更改

不再支持 Arguments.caller – IE9+

  • 从 IE9 开始，arguments.caller 属性在 IE9 标准模式或更高版本的文档模式中不受支持。
  • arguments.caller 属性在 IE5 (Quirks) 模式、IE7 标准模式和 IE8 标准模式中可以如期工作，但在 IE9 模式或更高版本的文档模式中无法正常运行。

不再支持传统 DOM 事件 – IE9+

  • 某些传统的文档对象模型 (DOM) 事件功能在 IE9 标准模式中已弃用，并且计划在下一个主要版本的最新标准模式中删除。 以下是弃用的事件以及你需要改用的基于标准的替代项：

不再支持条件注释 – IE10+

  • 为了提高与 HTML5 的可互操作性和兼容性，IE10 标准模式和 Quirks 模式中删除了对条件注释的支持。 这意味着，与在其他浏览器中相同，条件注释将被视作一般注释。
  • 使用了条件注释的页面在 IE9 中可正常工作，但在IE10 中无法正常工作。

不再支持 DX 筛选器 – IE10+

  • 对于“Internet 区域”中的网页，基于 DirectX 的筛选器和过渡（DX 筛选器）在IE10 中已过时。 默认情况下，这些功能在本地 Intranet 和受信任的站点区域中是禁用的，但可以针对所选文档模式进行启用。

不再支持 VML – IE10+

  • 对于IE10，VML 在下列文档模式中无法工作（已过时）：
  • 可互操作的 Quirks 模式
  • IE10 标准模式
  • 现在，VML 的普遍使用使得IE10 和当前版本的其他浏览器中实现了其相应的基于标准的备用项。IE的这一过时功能在IE10 提供的文档模式 5、7、8 和 9 中仍然可用，但这些模式下的性能逊色于它们在基于标准的硬件加速模式中的性能。

不再支持 XML 数据岛 – IE10+

  • 为了提高与 HTML5 的可互操作性和兼容性，IE10 标准模式和 Quirks 模式中删除了对“XML 数据岛”的支持。 这意味着，与在其他浏览器中相同，XML 数据岛将被分析为 HTML。

不再支持元素行为和 HTC – IE10+

  • 为了提高与 HTML5 的可互操作性和兼容性，IE10 标准模式和 Quirks 模式中删除了对元素行为和 HTML (HTC) 组件的支持。 这意味着，与在其他浏览器中相同，之前与元素行为或 HTC 绑定的元素都将被视作通用元素。

Internet 区域不再支持 CSS 表达式 – IE11+

  • 从 IE11 开始，对于在 Internet 区域中加载的网页，CSS 表达式在文档模式中不再有效。
    •  #fixed {
                            position: absolute;
                            left: 10px;
                            top: expression(body.scrollTop + 50 + "px");
                            background: white;
                            border: 1px solid red;
                        }

IE11 Internet区域边缘模式不再支持VBScript

  • 由于在 Internet 区域中，IE11 边缘模式不再支持 VBScript，因此网页中无法再使用以下 API 功能：
  • execScript 函数。
  • VBArray 对象。
  • "text/vbs" 和 "text/vbscript" MIME 类型（如同支持 script 元素的 type 值）。
